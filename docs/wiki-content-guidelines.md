# 트루핑키스 위키 지식 추가 지침

트루핑키스 위키는 학생이 교과 지식과 생활 지식을 안전하게 찾도록 돕는 교육용 위키입니다. 새 지식은 많아지는 것보다 정확하고 저작권 문제가 없게 쌓이는 것이 우선입니다.

## 0. 용어 기준

- 사용자가 읽고 찾는 개별 항목은 `지식`으로 부릅니다. 예: `지식 목록`, `오늘의 지식`, `함께 보면 좋은 지식`
- 출처로 확인한 원천은 `자료`로 부릅니다. 예: `확인한 자료`, `이 자료로 확인한 내용`
- 앱 내부 파일명과 코드 변수는 `documents.json`, `doc`처럼 기존 이름을 유지할 수 있습니다.
- `개념 지식`, `용어 지식`처럼 경계가 애매한 종류 구분은 사용하지 않습니다. 지식은 교과목, 분류, 태그, 검색 보조어, 함께 보면 좋은 지식으로 찾게 합니다.
- 외부 출처의 이름에 `문서`, `자료`, `보고서`가 들어간 경우에는 원래 제목을 바꾸지 않습니다.
- 새 화면 문구를 만들 때는 같은 기능을 `문서`, `지식`, `항목`으로 섞어 부르지 않습니다.

## 1. 출처 원칙

- 공공기관, 정부기관, 공공 교육기관, 박물관·과학관·국가기관 자료를 우선 사용합니다.
- 새 지식은 이용 조건상 참고·인용·재작성에 문제가 없는 자료에서만 확인합니다.
- 가져오거나 참고해도 되는지 확실하지 않은 자료는 사용하지 않습니다.
- 사이트에는 교육용 서비스이며 저작권자의 삭제 요청이 있을 경우 관련 게시물이 삭제될 수 있다는 안내를 표시합니다.
- 출처는 본문 사실을 확인하기 위한 자료입니다. 원문 문장, 이미지, 표, 그래프, 활동지, 도표를 그대로 가져오지 않습니다.
- 모든 문장에 외부 출처가 필요한 것은 아닙니다. 일반적인 배경 설명, 쉬운 비유, 예시, 퀴즈 문항은 AI와 작성자의 일반 지식으로 새로 쓸 수 있습니다.
- AI와 작성자의 일반 지식으로 새로 쓴 문장은 출처 항목으로 만들지 않습니다. 출처 항목에는 실제로 확인한 자료만 적습니다.
- 연도, 법령, 건강·안전, 금융, 역사적 사건, 특정 기관·인물처럼 틀리면 문제가 큰 사실은 공식 자료로 확인한 뒤 씁니다.
- AI는 작성 도구일 수 있지만 출처가 아닙니다. AI가 문장을 만들었다는 이유만으로 출처 항목을 만들지 않습니다.
- AI 도움으로 초안을 작성하거나 퀴즈·예시를 정리한 지식은 사이트 화면에서 AI 작성 도움 안내를 명확히 보여 줍니다.
- `sources`에는 실제로 사실 확인에 사용한 공식 자료, 공공 자료, 권위 자료만 적습니다.
- `usedFor`에는 자료 제목을 반복하지 말고 “이 자료로 무엇을 확인했는지”를 씁니다.
- 공공누리, 공공데이터, 정부·공공기관 자료라도 이용 조건을 확인하고 출처를 남깁니다.
- `All rights reserved`처럼 저작권 보호 표시가 있거나 재사용 조건이 분명하지 않은 공식 자료는 본문 사실 확인용으로만 참고합니다. 이 경우 원문 문장, 문단 구성, 표, 이미지, 그래프를 가져오지 않고 학생용 새 문장과 새 구조로 작성합니다.
- 상업 백과, 언론 기사, 민간 웹사이트는 보조 확인용으로만 제한적으로 사용합니다. 이용 조건이 불명확하면 새 지식의 핵심 출처로 쓰지 않습니다.
- 나무위키, 개인 블로그, 카페, 커뮤니티, SNS, 광고성 페이지, 출처 없는 요약문은 새 지식의 출처로 쓰지 않습니다.

## 2. 작성 방식

- AI 도움을 받을 수 있지만, 출처로 확인한 사실을 학생용 문장으로 새로 씁니다.
- 외부 자료를 참고하지 않고 일반 지식으로 설명을 작성할 때도 원문을 떠올려 바꿔 쓰는 방식이 아니라 처음부터 학생용 새 문장으로 씁니다.
- AI가 만든 설명, 예시, 퀴즈는 사람이 다시 읽고 어색한 표현, 억지 예시, 사실 확인이 필요한 문장을 점검합니다.
- 한 문장씩 바꿔 쓰는 수준의 베끼기를 하지 않습니다. 문단 구조와 예시도 새로 구성합니다.
- 초등 교과와 연결되는 개념은 쉬운 설명, 생활 예시, 헷갈리기 쉬운 점, 퀴즈를 함께 씁니다.
- 중학생이나 성인이 볼 수 있는 개념이라도 말투는 학생이 이해할 수 있게 유지합니다.
- 건강, 안전, 법, 금융, 개인정보처럼 중요한 주제는 “공식 자료와 보호자·선생님 확인이 필요하다”는 맥락을 넣습니다.
- 학생용 지식이라고 해서 모든 예시를 학교, 학급, 숙제, 친구 관계로 연결하지 않습니다.
- 학교폭력, 급식처럼 학교 맥락이 주제 자체인 경우를 제외하면 일상, 사회, 과학, 경제, 공공생활처럼 개념에 맞는 장면을 우선 씁니다.
- 예시는 억지로 친근하게 만들기보다 해당 개념을 정확히 이해하는 데 필요한 상황이어야 합니다.

## 3. 역사·인물·사건 추가 기준

- 개념 이해에 도움이 되는 경우에만 추가합니다.
- “처음 만든 사람”, “대표 인물”, “중요한 사건”은 한 사람의 업적으로 단정하지 않습니다.
- 논쟁이 있거나 정치적 해석이 갈리는 내용은 중립적으로 쓰고, 학생용 지식에 꼭 필요하지 않으면 제외합니다.
- 특정 개인이나 집단에게 피해가 될 수 있는 의혹, 평가, 비방성 표현은 넣지 않습니다.

## 4. 지식 구조

분류는 아래 역할을 나누어 씁니다.

- `subjects`: 교과 연결입니다. 화면에는 `관련 교과: 사회, 국어, 도덕`처럼 표시합니다.
- `categoryPaths`: 지식이 속한 분류 경로입니다. 한 지식이 여러 분류에 속할 수 있습니다. 예: `[["역사", "한국사", "조선"], ["과학", "탐구와 측정"]]`
- `mainTopic`: 대표 분류의 첫 칸입니다. 기존 화면과 검색 호환을 위해 유지합니다.
- `subTopicPath`: 대표 분류에서 `mainTopic` 뒤에 이어지는 경로입니다. 기존 화면과 검색 호환을 위해 유지합니다.
- `topicTags`: 여러 지식을 가로질러 묶는 태그입니다. 예: `역사 인물`, `훈민정음`, `문화`, `정치`

예시:

```json
{
  "subjects": ["사회", "국어", "도덕"],
  "categoryPaths": [
    ["역사", "한국사", "조선"],
    ["문화와 표현", "한글과 언어"]
  ],
  "mainTopic": "역사",
  "subTopicPath": ["한국사", "조선"],
  "topicTags": ["역사 인물", "훈민정음", "문화", "정치"]
}
```

카테고리는 지식의 위치를 정하고, 태그는 여러 지식을 연결합니다. `역사`, `한국사`, `조선`처럼 이미 분류 경로로 표시한 말은 태그에 반복해서 많이 넣지 않습니다.
경계에 걸친 지식은 `categoryPaths`에 여러 경로를 넣습니다. 예를 들어 장영실은 `역사 > 한국사 > 조선`과 `과학 > 탐구와 측정`에 동시에 속할 수 있습니다.

본문은 `chapters`에 큰 단원과 작은 단원으로 작성합니다. 권장 흐름은 아래와 같습니다.

1. 지식의 뜻과 핵심 기준
2. 생활 속 장면, 역사적 배경, 우리나라와의 연결, 실험·관찰처럼 지식에 맞는 전개
3. 오해하기 쉬운 부분이나 실제로 확인해야 할 기준
4. `퀴즈`

정보상자는 사용하지 않습니다. 지식 하단에는 `퀴즈`, `함께 보면 좋은 지식`, `확인한 자료`가 자연스럽게 이어져야 합니다.
관련 지식은 본문 목차 안에서 억지로 반복하지 않고, 하단의 `함께 보면 좋은 지식` 영역에서 연결합니다.
`쉽게 풀어보기`, `생활 속 예시`, `더 깊게 생각하기`, `다음 질문`처럼 어느 지식에나 붙일 수 있는 제목을 반복하지 않습니다.
`오해하기 쉬운 부분`, `뉴스에서 볼 때`, `생활 속 연결`처럼 너무 넓은 제목도 그대로 반복하지 말고, 해당 지식에서 무엇을 헷갈리기 쉬운지 드러나게 씁니다.
이미 뜻에서 충분히 쉽게 설명했다면 별도의 쉬운 풀이 단원을 만들지 말고, 예시·기준·오해·질문처럼 존재 이유가 분명한 단원으로 나눕니다.
목차는 지식마다 다를 수 있지만, 학생이 “왜 이 순서로 읽어야 하는지” 알 수 있어야 합니다.
각 작은 단원은 한 문장만 두고 끝내지 않습니다. 최소 2문단 이상으로 개념, 기준, 예시, 주의점 중 하나를 분명히 설명합니다.
한 지식 전체는 최소한 뜻, 원리나 배경, 생활·교과 예시, 오해 바로잡기를 포함해 충분한 본문량을 갖춥니다.
`퀴즈`는 객관식이어야 하며, 문제마다 질문, 보기 4개, 정답 번호, 해설을 넣습니다.
질문은 무엇을 묻는지 분명해야 합니다. `뜻`, `영향`, `결과`, `활동`, `주의해서 기억할 점`처럼 답을 고르는 기준을 문제 문장 안에 직접 씁니다.
`다음 보기 중`, `본문 내용과 맞는 문장`, `특징이나 예시`처럼 생성된 문제처럼 보이는 표현을 반복하지 않습니다.
`설명으로 맞는 것은?`, `조심해야 할 점은?`처럼 범위가 넓어 보기를 읽기 전까지 무엇을 묻는지 알기 어려운 질문을 반복하지 않습니다.
좋은 예시는 `급식과 영양에 대해 알맞게 설명한 것은 무엇인가요?`, `6·25전쟁의 결과로 알맞은 것은 무엇인가요?`, `세종대왕의 업적으로 알맞은 것은 무엇인가요?`처럼 자연스러운 교실 문제 문장입니다.
오답 보기는 다른 지식의 정의를 가져와 채우지 말고, 해당 지식에 대한 그럴듯하지만 틀린 설명으로 씁니다.
예를 들어 범위를 너무 좁게 보거나, 원인과 결과를 뒤집거나, 중요한 조건과 예외를 빠뜨린 문장으로 구성합니다.
누가 봐도 관련 없는 보기나 모든 지식에 반복되는 범용 오답 문구는 사용하지 않습니다.
오답 보기임이 바로 드러나는 어색한 설명형 문장이나 `무조건`, `언제나`, `항상` 같은 단정 표현을 반복하지 않습니다.
해설은 “본문에서는 ~라고 설명합니다” 같은 범용 문구로 끝내지 말고, 왜 그 선택지가 맞는지와 어떤 기준을 함께 봐야 하는지까지 설명합니다.
한 지식 안에서 여러 문제가 같은 정답 문장을 반복하지 않도록 합니다.
퀴즈 정답 문장은 현재 본문이나 정의에서 확인할 수 있어야 하며, 본문을 고치면 퀴즈도 함께 확인합니다.

## 5. 출처 표기 형식

`sources`에는 문자열을 넣지 않고 아래 형식의 객체만 넣습니다.

```json
{
  "publisher": "기관명",
  "title": "자료명",
  "url": "https://example.go.kr/page",
  "usedFor": "이 자료로 확인한 내용",
  "license": "공식 자료 확인 후 학생용으로 새로 작성, 원문/이미지/표/그래프 복제 없음",
  "checkedAt": "2026-05-14"
}
```

`copyrightNote`에는 지식이 원문을 복제하지 않았고 학생용 문장으로 새로 작성되었다는 설명을 넣습니다.

## 6. 새 지식 추가 체크리스트

- 공식 자료 또는 이용 조건이 분명한 자료를 확인했는가?
- 출처 URL, 확인한 내용, 이용 메모, 확인일을 모두 적었는가?
- 원문 문장, 이미지, 표, 그래프를 가져오지 않았는가?
- 학생이 이해하기 쉬운 설명과 예시가 있는가?
- 제목, 본문, 교과, 분류, 태그, 검색 보조어로 검색될 수 있는가?
- 함께 보면 좋은 지식이 실제 존재하는 지식 ID로 연결되는가?
- 정보상자 없이 큰 단원, 작은 단원, 퀴즈 구조로 정리되었는가?
- `node scripts/validate-wiki-content.mjs`를 통과했는가?
